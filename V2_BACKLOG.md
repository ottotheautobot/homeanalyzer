- ~~**Floor-plan reconstruction polish.**~~ Shipped v1.6: curope CUDA kernel built (~2× speedup), Manhattan global alignment, adaptive DBSCAN eps based on tour pace. v1.7/v1.8 added a heuristic stack (view-association filter, overlap merge with centroid guard, door cap, dimension-based confidence downgrade). Wins on short tours; still open for long multi-room tours where the camera-cluster-of-3D-points approach hits a structural ceiling — see new "true room segmentation" backlog item below. Feasibility detail in `docs/floorplan-slam-research.md`.
- **[P4] True room segmentation, not camera-cluster heuristics.** The v1.6–v1.8 stack runs DBSCAN on camera positions, then for each cluster takes the bbox of wall-height points its frames saw. Heuristic ceiling: when the camera traverses adjacent rooms, multiple clusters legitimately see the same shared walls — so wall-leakage is not a filterable noise term, it's a feature of the input. Tightening the view-association radius (5m → 3m) shrinks bboxes by 10-20% but doesn't separate rooms. Multi-room tours (Cooper City 24-min, Savannah Circle 18-min) consistently collapse to 1-2 oversized "rooms". Real fixes need a different segmentation primitive: (1) detect doorways from point-cloud wall-gap geometry, then split on those instead of DBSCAN-on-cameras, (2) use a purpose-built room-segmentation model like RoomFormer that takes a global point cloud + outputs room polygons, (3) sequential temporal segmentation (camera dwell-time + motion patterns) — fundamentally different signal from spatial clustering. Each is a multi-day project, not a heuristic tweak.

## Research / exploration

- ~~**Compare Sonnet 4.6 vs Opus 4.7 for synthesis quality.**~~ Ran 2026-04-28 against 3 long real tours (Cooper, Savannah, Lake Dasha). Verdict: stay on Sonnet 4.6. Quality is comparable — both models surface the same top concerns, deal-breakers, and highlights. Opus was actually faster on all three (30-37s vs 34-45s) but ~5x more expensive per call ($0.12 → ~$0.60). Score calibration differs (Opus +0.7 / +1.0 on two houses, same on one) — Opus is more lenient. The 5x cost increase isn't justified by the quality delta and would re-baseline cross-house score comparability. Output saved to `/tmp/synthesis_ab.md` for posterity.
- ~~**Explore Claude's prompt caching with 1-hour TTL.**~~ Shipped 2026-04-28: `extract.py`, `vision.py` (batch path), `compare.py` cache_control blocks now request `ttl: "1h"`. The transcript-extraction path (~20s cadence during a live tour) is the highest-value site — 5min default was missing on slower tours.
- **[P5] Whisper local vs. cloud transcription.** Currently using OpenAI Whisper API for async path. mlx-whisper on Apple Silicon is free and fast — could be a self-hosted user option.
- **[P1] Verify MB video archive isn't degraded by Zoom's active-speaker bandwidth shifts.** User report 2026-04-28: a remote viewer's live Zoom video froze when another remote participant was speaking. That's classic Zoom adaptive-bitrate behavior favoring the active speaker. Open question: does this also degrade the **archived mp4 that MB uploads to us**, or does MB capture each participant's stream independently at full quality? If the archive is fine, the symptom is purely a viewing-experience issue (and a separate motivation for owning capture); if the archive is also degraded, that's a quality bug affecting every multi-participant tour and bumps the streaming-in-app item up the priority list. Repro: stage a 3-person Zoom tour, have one remote participant talk continuously while the host walks the home, then inspect the resulting MB mp4 frame-by-frame for dropped/low-quality segments around speech boundaries.
