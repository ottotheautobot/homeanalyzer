- **[P4] True room segmentation, not camera-cluster heuristics.** The v1.6–v1.8 stack runs DBSCAN on camera positions, then for each cluster takes the bbox of wall-height points its frames saw. Heuristic ceiling: when the camera traverses adjacent rooms, multiple clusters legitimately see the same shared walls — so wall-leakage is not a filterable noise term, it's a feature of the input. Tightening the view-association radius (5m → 3m) shrinks bboxes by 10-20% but doesn't separate rooms. Multi-room tours (Cooper City 24-min, Savannah Circle 18-min) consistently collapse to 1-2 oversized "rooms". Real fixes need a different segmentation primitive: (1) detect doorways from point-cloud wall-gap geometry, then split on those instead of DBSCAN-on-cameras, (2) use a purpose-built room-segmentation model like RoomFormer that takes a global point cloud + outputs room polygons, (3) sequential temporal segmentation (camera dwell-time + motion patterns) — fundamentally different signal from spatial clustering. Each is a multi-day project, not a heuristic tweak.

## Research / exploration

- ~~**Compare Sonnet 4.6 vs Opus 4.7 for synthesis quality.**~~ Ran 2026-04-28 against 3 long real tours (Cooper, Savannah, Lake Dasha). Verdict: stay on Sonnet 4.6. Quality is comparable — both models surface the same top concerns, deal-breakers, and highlights. Opus was actually faster on all three (30-37s vs 34-45s) but ~5x more expensive per call ($0.12 → ~$0.60). Score calibration differs (Opus +0.7 / +1.0 on two houses, same on one) — Opus is more lenient. The 5x cost increase isn't justified by the quality delta and would re-baseline cross-house score comparability. Output saved to `/tmp/synthesis_ab.md` for posterity.
- ~~**Explore Claude's prompt caching with 1-hour TTL.**~~ Shipped 2026-04-28: `extract.py`, `vision.py` (batch path), `compare.py` cache_control blocks now request `ttl: "1h"`. The transcript-extraction path (~20s cadence during a live tour) is the highest-value site — 5min default was missing on slower tours.
- **[P5] Whisper local vs. cloud transcription.** Currently using OpenAI Whisper API for async path. mlx-whisper on Apple Silicon is free and fast — could be a self-hosted user option.
- **[P2] Verify MB video archive isn't degraded by Zoom's active-speaker bandwidth shifts.** User report 2026-04-28: a remote viewer's live Zoom video froze when another remote participant was speaking. That's classic Zoom adaptive-bitrate behavior favoring the active speaker. Open question: does this also degrade the **archived mp4 that MB uploads to us**, or does MB capture each participant's stream independently at full quality? If the archive is fine, the symptom is purely a viewing-experience issue (and a separate motivation for owning capture); if the archive is also degraded, that's a quality bug affecting every multi-participant tour and bumps the streaming-in-app item up the priority list. Repro: stage a 3-person Zoom tour, have one remote participant talk continuously while the host walks the home, then inspect the resulting MB mp4 frame-by-frame for dropped/low-quality segments around speech boundaries. Tabled 2026-04-28 — needs a 3-person test tour to repro.
- **[P4] Capture-vs-collaboration architectural split.** The current architecture conflates two jobs into one transport (Zoom + MB): high-fidelity capture for the analysis pipeline AND live sharing with a remote follower. Splitting them lets each be optimized independently: (a) **capture** — local 4K (and potentially LiDAR-derived depth) recorded on-device, decoupled from any live link, fed to floor-plan reconstruction at full quality with no codec/bandwidth losses; (b) **collaboration** — a low-bandwidth audio stream + the live observation feed as the partner's UX, replacing video mirroring entirely. Eliminates Meeting BaaS as a transport dependency, materially improves analysis input quality, and may produce a *richer* follower UX than passively watching shaky video. Investigate **after** customer discovery validates whether the buyer/inspector use case rewards capture quality enough to justify the iOS-app cost. Heavy overlap with the line 47 "replace the Zoom-bot middleman" item — this is the cleaner architectural re-framing of the same problem.
- **[P4] Native iOS capture app (architectural prerequisite).** Pointer to the existing P4 native-iOS item (Vendor / infrastructure section). If the capture-vs-collaboration split is committed, the iOS app stops being a "nice to have" for camera/audio quality and becomes a hard prerequisite — neither PWA nor web has access to 4K video capture or LiDAR on iOS Safari. Tracking here so the discovery-phase decision sees the architectural blocker, not just the productization argument.

## Inspector vertical

Items specific to the home-inspector customer path being explored in customer discovery. None of these justify engineering work until at least one inspector validates the use case in a discovery call. Listed here so the path is concrete enough to discuss.

- **[P4] Adapt extraction prompts for the inspection use case.** The current prompts are buyer-tour-shaped (concerns, deal-breakers, partner reactions). Inspector context needs a defect taxonomy aligned to ASHI / InterNACHI Standards of Practice, severity tied to standardized ratings (safety / major / minor / monitor), and state-specific compliance fields where required. Same pipeline, different tool schema + system prompt. Effort is days, not weeks.
- **[P4] PDF report generation aligned to standard inspection formats.** Inspectors deliver structured reports, not synthesis briefs. Need a templated PDF (HomeGauge / Spectora / Horizon are the dominant formats to model on) with per-system organization, photo-defect linking, summary + recommendations. Deliverable shape matters for inspectors more than for buyers.
- **[P4] Higher accuracy bar — human-in-the-loop confirmation for safety-related observations.** Buyer-side false positives are tolerable; inspector-side false positives are liability. Any observation flagged as a safety issue should require explicit inspector confirmation before landing in the report. Affects the extraction → review UI.
- **[P4] Inspector-specific UI: per-system organization + photo-defect linking.** Buyer UI groups by room. Inspector UI groups by system (electrical, plumbing, HVAC, structural, exterior, roofing, etc.) with each defect linked to one or more reference photos. Same `observations` table, different default grouping + a `system` field added.
