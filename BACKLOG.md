## Post-Zoom architecture: in-app capture-and-distribute

The candidate v2-architecture replacement for the Zoom + Meeting BaaS transport. Not committed; gated on customer discovery validating that capture quality and collaboration UX matter enough to justify the build. Lives as a single rich entry below because the components are interlocking â€” picking one part doesn't make sense without the others.

- **[P4] Native iOS app that owns capture + distribution end-to-end.** One iPhone app runs three concurrent streams from a single camera/mic source: (1) **full-quality local recording** (4K + uncompressed audio, written to device, uploaded post-tour) feeds the floor-plan + vision analysis pipelines at full fidelity; (2) **downscaled live video** (~720p adaptive) streamed to followers via WebRTC; (3) **live audio** streamed to followers as a separate track so it survives video-bandwidth drops. Replaces Zoom + Meeting BaaS as the transport for both capture and collaboration.
  - **Follower UX is the existing observation feed + live video + audio**, not a passive Zoom mirror. Web-based (LiveKit web SDK + existing Next.js surfaces); no second native app needed initially.
  - **Push-to-talk followerâ†’host audio.** Avoids the echo problem that any full-duplex setup hits (host plays partner's voice through phone speaker â†’ analysis mic re-captures it â†’ polluted transcript). Partner presses-and-holds to ask a question; host hears it through earbuds; host responds verbally and that's already in the analysis stream. PTT covers ~80% of what tour-collab actually needs (asking, reacting, narrating) without the architecture pain of full-duplex.
  - **WebRTC infra: LiveKit Cloud as the SFU.** Free tier covers discovery-phase usage (10K participant-minutes/mo). Migrate to self-host LiveKit only if usage justifies. Daily.co and self-hosted mediasoup are alternatives but worse for dev velocity at this stage.
  - **Apple Developer account** ($99/yr) is the only external blocker. TestFlight covers first 100 testers â€” no App Store review needed for discovery.
  - **What this fixes vs. today's Zoom path:** capture quality (no codec/bandwidth losses on the analysis input), partner UX (predictable video regardless of who's speaking â€” no Zoom active-speaker bandwidth shifts), interactive capture protocol becomes possible ("hold here for 3 seconds"), eliminates ~$0.85/hr Meeting BaaS spend, eliminates Zoom Pro upgrade pressure, eliminates OBF token / per-meeting waiting-room friction.
  - **What this doesn't fix:** the host still has to keep the phone unlocked and the app foregrounded during the tour (acceptable â€” it's an active walking workflow), and we lose Zoom as the seller-agent's "I can see what's being recorded" verification surface (low-cost mitigation: explicit consent UI on the host phone before recording starts).
  - **Effort estimate:** 2-3 weeks for the iOS capture flow (LiveKit + local-record + upload), plus 3-5 days for the follower web view. Backend changes minimal â€” uploaded mp4s land in the existing Supabase Storage path. Don't start until discovery validates the bet.
  - **Open questions for the discovery phase:** does 4K-locally-captured-then-uploaded analysis output noticeably beat what we get from Zoom-passed-through video today? Is push-to-talk-as-collab acceptable to followers, or does it feel awkward versus Zoom's continuous video? How important is LiDAR depth (iPhone Pro only) for floor-plan reconstruction quality vs. monocular-only at full resolution?
  - **Visual floor plan as a real differentiator (CubiCasa-tier).** Once the in-app capture path lands and we can interject "slow pan in this room, pause at corners" prompts to influence what the camera sees, a *good* visual floor plan becomes possible â€” and would be a genuine product differentiator that the v1-v2.6 measured pipeline never reached. Today's approach (DBSCAN on cameras + concave hull of wall-height points) hits a structural ceiling regardless of input quality (validated 2026-04-29 against locally-recorded 4-min tour: same 1-2 oversized rooms as Zoom-passed-through). A real solution likely needs RoomFormer-style learned segmentation or doorway-aware splitting â€” see the gated P4 "true room segmentation" entry above.
